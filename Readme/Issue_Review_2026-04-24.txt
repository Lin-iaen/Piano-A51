【发生了什么问题？】

  现象：
    - 上位机正常发送音符帧
    - 板子上的 LED 有反应（说明串口确实收到了数据）
    - 但蜂鸣器偶尔不响，或者响得不对

【为什么？—— 根因分析】

  问题出在"串口中断"和"主循环"同时访问同一块数据。

  场景回放：
    1. 上位机发了一帧 8 个字节的音符数据
    2. UART中断每收到一个字节就把数据写入 UART_RxBuf[]
    3. 主循环同时去读 UART_RxBuf[] 做协议解析
    4. 假设主循环读到第3个字节时，第4个字节又到了——
       中断抢走CPU，把第4字节覆盖进缓冲区，
       主循环恢复后读到的是"新旧拼凑"的半帧数据
    → 解析出来的音符值可能是乱码，蜂鸣器自然不正常

  这就是经典的"竞态条件"（Race Condition）问题。
  在多任务系统用锁/信号量解决，在裸机单片机里用"关中断"解决。

【怎么修改的？】

  修改了 2 个文件：

  1. Drv_Uart.asm — 增加了 8 状态帧接收机
     之前：简单地把收到的字节往缓冲区塞
     修改后：逐字节判断是帧头/命令/长度/数据/CRC，
     任何一步不匹配就丢弃整帧从头等。
     效果：宁可不收，不收错的。杂散噪音不再伪造出"假数据"。

  2. Protocol.asm — 增加了"原子快照"读取
     之前：直接读 UART_RxBuf（边收边读，必乱）
     修改后：
       CLR ES       ← ①先关串口中断（禁止新数据进来）
       快照 Cmd/Len/Payload → 复制到局部变量
       UART_RxReady = 0
       SETB ES      ← ②再开中断
       用局部变量做后续解析
     效果：读到的一定是一帧完整、一致的数据。

【学到的经验】

  1. 中断和主循环共享变量时，主循环读取前必须关中断
  2. 串口接收要做帧状态机，不能"来什么存什么"
  3. 缓冲区长度要设上限保护（这里设为8字节）
  4. 这些坑在 C 语言里同样存在，原理是通用的
